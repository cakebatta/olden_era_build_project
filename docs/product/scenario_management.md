# Scenario Management Product Specification

## Purpose

This document defines the user-facing behavior of persistent planning scenarios. A saved scenario is a named, persistent planning workspace containing user intent. It is not a cached analysis result.

The specification freezes product behavior for the first persistence implementation. Toolkit-specific dialogs, widget layout, and visual styling are outside scope.

## Product definition

A scenario contains:

- a stable scenario identity and user-visible name;
- one faction;
- one selected target building SID and level;
- one starting game date;
- starting-building overrides represented by `PlanningScenario`;
- starting resources;
- user-authored recruitment actions;
- a description and notes.

The application regenerates all analysis from these inputs, current canonical game data, and current backend rules.

## Scenario states

### New unsaved scenario

A new unsaved scenario has an in-memory UUID and timestamps but has not been written to the managed repository. It is dirty because no saved representation exists. Creation does not automatically save.

### Saved clean scenario

A saved clean scenario exists in the managed repository and has no user-authored changes relative to its most recently loaded or successfully saved representation.

### Saved modified scenario

A saved modified scenario exists in the managed repository but its current user-authored state differs from the clean baseline. It is dirty.

### Loaded scenario

A loaded scenario is retrieved from the managed repository and made active. Loading preserves identity, metadata, and user-authored content, establishes a clean baseline, does not change timestamps, and does not itself generate analysis.

### Duplicated scenario

A duplicated scenario copies user-authored content, receives a new UUID and new timestamps, and becomes a distinct logical scenario. It is initially unsaved and dirty. Generated analysis is not copied.

### Imported scenario

An imported scenario is an external portable document that has passed validation and has deliberately been added to the managed repository. Version 1 exposes import-as-copy: it preserves user-authored content but generates a new UUID and new timestamps.

### Deleted scenario

A deleted scenario no longer exists in the managed repository. Deletion is by UUID, requires confirmation, removes only the managed document, and does not affect exported copies or other scenarios.

## Dirty-state semantics

The desktop must track dirty state.

A scenario becomes dirty whenever persisted user-authored state differs from the most recently loaded or successfully saved semantic representation.

Changes that make a scenario dirty include:

- rename, description, or notes changes;
- faction, target SID, target level, or starting-date changes;
- starting-building override changes;
- starting-resource changes;
- recruitment action additions, removals, edits, or reordering.

The following do not make a scenario dirty:

- generating or regenerating plans, income, stock, ledgers, comparisons, or Decision Summaries;
- selecting, sorting, or filtering generated results;
- scrolling, resizing, highlighting rows, or opening panels;
- navigating between desktop workspaces.

The clean baseline is semantic document content, not the original file's whitespace or key order.

## Identity and metadata

Every persisted scenario has:

- a stable UUID;
- a user-visible name;
- a description;
- creation and modification timestamps;
- a schema version.

The UUID remains unchanged when the user edits, renames, saves, exports, closes, reopens, or loads the scenario.

A new UUID is generated by Create, Duplicate, Save As, and import-as-copy.

## Naming rules

Scenario names:

- must be strings;
- are normalized by removing leading and trailing whitespace;
- must contain at least one non-whitespace character;
- may contain internal repeated whitespace;
- may duplicate another scenario's display name;
- have a maximum normalized length of 120 Unicode code points.

Duplicate names are allowed because UUID is authoritative. Display names never determine repository filenames or storage identity.

## Operations

### Create

1. Resolve active unsaved work using Save, Discard, or Cancel.
2. Construct a new document with a new UUID and current UTC timestamps.
3. Apply Version 1 defaults.
4. Make it active and dirty.
5. Do not write it until Save succeeds.

### Open

1. Resolve active unsaved work using Save, Discard, or Cancel.
2. Select a managed scenario by UUID.
3. Retrieve and validate it.
4. Perform canonical-data validation before analysis.
5. Make it active and clean.
6. Preserve identity and timestamps.
7. Do not generate analysis as part of repository retrieval.

A document with unresolved canonical references may be inspectable, but analysis must be blocked and the document must not be silently rewritten.

### Save

For an existing managed scenario, Save updates the matching UUID-owned repository entry. It preserves `created_at` and updates `modified_at` only after a successful content save.

For a new unsaved scenario, Save performs its first managed save.

A successful Save establishes the saved document as clean. A failed Save leaves the active scenario dirty and preserves the prior canonical repository file. A stale save must not silently overwrite newer content.

### Save As

Save As creates a new logical scenario from the active user-authored content:

- generate a new UUID;
- require a valid name;
- generate new creation and modification timestamps;
- save to a new UUID-owned location;
- make the new scenario active and clean;
- leave the source unchanged.

### Rename

Rename changes only the display name. It preserves UUID and `created_at`, marks the scenario dirty, and updates `modified_at` when the rename is successfully saved. It does not rename the UUID-owned file.

### Duplicate

Duplicate copies all current user-authored content, including deliberate unsaved edits, but not analysis. It generates a new UUID and timestamps, proposes or requests a valid name, and creates a new unsaved dirty scenario. The source is unchanged.

### Delete

Delete removes a managed scenario by UUID after confirmation. If the active scenario has unsaved work, resolve it first using Save, Discard, or Cancel. Deleting a non-active scenario does not alter the active scenario.

### Import

Import adds an external portable JSON document to the managed repository.

1. Select an external file.
2. Enforce file-safety limits.
3. Detect version, validate structure, construct domain values, and migrate only when an explicit migration exists.
4. Perform canonical-data validation before analysis.
5. Modify the repository only after validation and collision policy succeed.

UUID collision policy:

- never silently overwrite an existing UUID;
- Version 1 exposes import-as-copy;
- import-as-copy creates a new UUID and timestamps while preserving user-authored planning content;
- replacement is deferred to a future explicit workflow;
- ordinary Save is the operation that updates an existing managed UUID.

### Export

Export writes the last successfully saved managed document to a user-selected destination.

It:

- preserves UUID, name, schema version, and timestamps;
- does not change repository location or active identity;
- does not make the scenario dirty;
- does not update `modified_at`;
- requires explicit confirmation before overwriting a destination.

Version 1 does not export unsaved edits; the user must save first.

## Unsaved-work policy

Whenever an operation would lose active changes, present:

- **Save** — save, then continue only if saving succeeds;
- **Discard** — abandon changes, then continue;
- **Cancel** — leave active scenario and repository unchanged.

This applies when opening another scenario, creating a scenario, closing the application, or deleting the active scenario.

## Destructive-operation expectations

Explicit confirmation is required for:

- deleting a saved scenario;
- discarding unsaved changes;
- overwriting an existing export file.

Messages must identify the affected scenario or destination and must not imply recovery because Version 1 has no trash or revision history.

## Persistent versus transient workspace state

Version 1 persists faction, selected target SID and level, and starting date because they define the intended analysis.

Version 1 does not persist highlighted rows, scroll position, window size, selected generated plan, open panels, active tab, transient error text, or formatting choices.

## Analysis boundary

Loading supplies user intent. It does not create dependency graphs, build orders, `BuildPlan`, `IncomeTimeline`, `RecruitmentStock`, `ResourceLedger`, comparisons, Decision Summaries, or formatted rows.

The desktop submits loaded intent to the authoritative Query Layer.

## Out of scope

Version 1 does not define autosave, cloud sync, Steam Cloud, accounts, collaboration, revision history, automatic merge, encryption, binary formats, result caching, tags, folders, templates, or UI layout.
